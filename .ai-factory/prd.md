# OurMap (ourmap)

# PRD
## Background / Problem
- 첨부 사진은 **장당 최대 10MB**까지 지원하는 수준으로 “사진 기반 기록”이 중심이며, 텍스트 입력 부담을 줄이는 방향이 전제되어 있다.
- 국내 좌표 정확도가 높은 **카카오 로컬 API(역지오코딩)** 는 무료 한도 **일 30만 건**으로 MVP에서 장소명 자동 추정에 현실적으로 활용 가능하다.

데이트/여행 사진이 휴대폰 갤러리에 흩어져 있어, 시간이 지난 뒤 “그때 그 식당/카페가 어디였지?”를 다시 찾기 어렵다. 기존 여행 기록 앱(예: Polarsteps, Wanderlog)은 SNS 공유 성격이 강하거나 사용 흐름이 무거워 일상 데이트 기록에는 부담이 크다. 커플 **둘만 보는 비공개** 전용으로, 사진만으로 빠르게 장소를 쌓고 지도에서 회상할 수 있는 도구가 부족하다.

## Goal (1-sentence product definition)
EXIF GPS가 포함된 사진 기준으로 사용자가 **장소 1건을 30초 이내에 기록**하고, **6개월 후에도 지도/여행 목록에서 1초 이내에 해당 기록을 찾을 수 있는** 커플 전용 비공개 장소 아카이브를 제공한다.

## Non-goals
- 외부 공유, SNS 피드, 댓글, 좋아요 등 **소셜 기능**
- 가본 적 없는 곳 추천/탐색 등 **추천 시스템**
- 실시간 동시 편집 등 **멀티유저 협업**(v1은 단일 공유 계정 전제)
- 자동 클러스터링, 태그, 회상 알림, 결산 카드 등 **백로그 기능**
- **오프라인에서 신규 장소 입력 후 동기화**(v1.5+ 검토)

## Target Users (personas + use cases)
- 김민준(28, 직장인) — 역할: 데이트 기록 담당  
  - Pain point: “그때 그 카페”를 자주 잊어버려 갤러리에서 사진을 뒤지며 시간을 낭비함. 기록 앱은 입력이 귀찮아 포기.
  - Use case: 데이트 직후 사진만 선택해 위치/날짜 자동으로 채우고 30초 내 저장.
- 이서연(27, 직장인) — 역할: 여행 추억 정리 담당  
  - Pain point: 블로그/SNS에 올리기엔 사적이고 번거로움. 둘만 보는 가벼운 정리가 필요.
  - Use case: 여행 후 장소들을 여행 단위로 묶어 지도+리스트로 회상.

## Target Market
- South Korea (KR), 20–30대 커플 중심의 사적 기록 니즈(데이트/국내 여행 빈도가 높은 모바일 사용자)

## Data Entities (nouns with key fields)
- **Place**
  - id
  - name
  - latitude, longitude
  - address (역지오코딩 결과/사용자 수정)
  - visited_at (방문일=사진 촬영일 기본)
  - review (한줄평)
  - rating (1–5)
  - category (맛집/카페/숙소/명소/액티비티/기타)
  - trip_id (nullable)
  - created_at, updated_at
- **Photo**
  - id
  - place_id
  - storage_url
  - thumb_url
  - taken_at (EXIF)
  - sort_order
  - file_size_kb (또는 bytes; 업로드 제한 검증용)
- **Trip**
  - id
  - name
  - start_date, end_date
  - cover_photo_url (optional)
  - description (optional)
  - created_at, updated_at

## Core Flow (numbered steps)
1. 사용자는 앱에서 로그인 후 탭 내비게이션으로 진입한다.
2. **리스트 탭** 우하단 `+` 버튼을 탭해 **장소 추가**를 시작한다.
3. 갤러리에서 사진(여러 장) 선택 → 앱이 **EXIF에서 촬영일/좌표를 자동 추출**한다.
4. 지도에서 핀 위치를 확인하고, 좌표가 없거나 부정확하면 **핀 드래그로 위치를 보정**한다.
5. 카카오 로컬 API 역지오코딩으로 추정된 **장소명**을 확인/수정하고, 한줄평/별점/카테고리를 입력한다.
6. 저장하면 **리스트/지도에 즉시 반영**되며, 사용자는 **여행 탭에서 해당 장소를 여행에 연결/해제**할 수 있다.

## Success Metrics (measurable)
- EXIF GPS가 있는 사진 기준 **장소 1건 기록 소요 시간 ≤ 30초**
- EXIF GPS 보유 사진에서 **핀 자동 표시 성공률 ≥ 95%**
- 장소 20건 이상에서 **리스트/지도 초기 로딩 ≤ 1초(LCP 기준)**
- 6개월 후 **누적 장소 기록 ≥ 50건**
- 여행 생성 후 **장소 연결까지 완료율 ≥ 80%**

## MVP Scope (exhaustive feature list)
- **장소 기록**: 사진 여러 장 첨부(장당 ≤10MB), EXIF로 GPS/촬영일 자동 추출, 장소명/한줄평/별점/카테고리 입력
- **위치 보정**: EXIF GPS 없음/오차 시 지도에서 핀 드래그로 수정, 카카오 로컬 API로 역지오코딩하여 장소명 자동 추정값 제공
- **리스트 뷰**: 방문일 역순 카드(대표 사진 + 장소명 + 한줄평 + 별점) 목록
- **지도 뷰**: 전체 장소 핀 표시, 핀 탭 시 요약 팝업, 상세 화면으로 이동
- **여행 묶음(Trip)**: 여행 생성/편집, 장소의 여행 연결/해제, 여행 상세에서 소속 장소 리스트+지도
- **장소 상세/수정/삭제**: 사진 갤러리, 메타데이터(한줄평/별점/카테고리/방문일/소속 여행) 확인 및 편집, 삭제

**Mobile-specific UX 요구사항(범위 내 동작 정의)**
- **갤러리 접근**: 사진 선택은 기기 **Photo Library 권한**을 요청하며, 권한 거부 시 “설정에서 권한 허용” 안내 화면/문구를 노출한다(카메라 촬영 플로우는 MVP 제외).
- **Pull-to-refresh 적용 화면**
  - 리스트 탭(`/list`): 당겨서 새로고침 시 서버에서 Place 목록 재조회
  - 지도 탭(`/map`): 당겨서 새로고침 시 Place 목록 재조회 후 핀 리렌더
  - 여행 목록(`/trips`): 당겨서 새로고침 시 Trip 목록 재조회
  - 여행 상세(`/trips/[id]`): 당겨서 새로고침 시 Trip + 소속 Place 재조회
- **제스처**
  - 리스트 카드: 카드 **스와이프(좌→우 또는 우→좌 중 1방향 고정)** 로 “삭제” 액션 노출(삭제 확인 다이얼로그 포함)
  - 지도: 핀 탭으로 요약 팝업, 팝업 탭으로 상세 이동(롱프레스 등 추가 제스처는 MVP 제외)
  - 장소 추가/편집 지도: 핀을 **드래그**하여 좌표 수정(드래그 종료 시 역지오코딩 재호출)
- **오프라인 고려(제한 명시)**: MVP는 오프라인 입력/동기화를 제공하지 않으며, 네트워크 오류 시 재시도 CTA와 오류 메시지를 표시한다.

## Target Audience & Marketing
- 타겟: SNS 공유는 부담스럽지만 데이트/여행 기록은 남기고 싶은 **20–30대 커플**
- 핵심 가치 제안: “**사진만 올리면 끝, 우리 둘만 보는 장소 일기**”
- 핵심 메시지(랜딩/스토어): 사진→자동 위치 기록 / 지도로 보는 발자국 / 여행 단위로 묶는 추억 앨범
- 채널: 카카오톡, 네이버 블로그, 유튜브(한국)

## Monetization Strategy
- **freemium**  
  - 이유: 커플의 지속 사용(장소 누적, 여행 묶음)이 가치로 쌓이는 유형이라 무료 진입장벽을 낮추고, 향후 저장 용량/여행 개수/고급 정리 기능 등으로 업셀 구조가 적합하다.  
  - 단, **MVP에서는 인앱 결제/구독 구현은 포함하지 않는다**(향후 버전에서 App Store/Google Play 결제로 확장).

## Assumptions
1. v1은 커플이 **공유 계정 1개**로 사용한다.
2. 사용자는 주로 **갤러리 사진 선택**으로 장소를 기록한다(카메라 촬영은 MVP 범위 밖).
3. EXIF GPS가 없는 사진도 일정 비율 존재하며, 이 경우 사용자는 지도에서 핀 드래그 보정을 수행할 의사가 있다.
4. 카카오 로컬 API 호출을 위한 키/쿼터는 MVP 운영 범위에서 확보 가능하다.
5. 사진 업로드 제한(장당 10MB)은 사용자 불만을 크게 유발하지 않는 수준이다.
6. 장소는 기본적으로 visited 상태만 사용한다(위시리스트 UI는 제외).
7. 지도/리스트에서 “1초 내 탐색” 목표는 데이터 20건 이상에서도 유지 가능하도록 구현/최적화가 필요하다.
8. 개인정보/프라이버시 기대치가 높으므로 외부 공유 기능이 없는 것이 오히려 핵심 가치로 작동한다.
